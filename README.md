# Provenance Guard

A backend classification API that detects whether submitted text was written by a human or generated by AI. Built for creative platforms where attribution transparency matters.

---

## Architecture Overview

A submitted piece of text travels through five components before a label reaches the user:

1. **Rate Limiter** - Flask-Limiter checks the request against per-IP limits. Requests over the limit receive a 429 before touching any detection logic.
2. **Detection Pipeline** - Two independent signals run in sequence: the LLM-based classifier (Groq) and the stylometric analyzer (pure Python). Each produces a score from 0 to 1.
3. **Confidence Scorer** - Combines both signals using a weighted average (60% LLM, 40% stylometric) into a single `combined_score`, then maps it to one of three attributions using asymmetric thresholds.
4. **Label Generator** - Maps the attribution and confidence to one of three verbatim transparency label variants written in plain language for non-technical readers.
5. **Audit Logger** - Every classification decision and every appeal is written to a structured JSON log before the response is returned.

```
SUBMISSION FLOW
---------------
Client
  |
  |  POST /submit  {text, creator_id}
  v
Rate Limiter --429--> Client
  |
  v
Detection Pipeline
  |-- Signal 1: Groq LLM      --> llm_score (0-1)
  |-- Signal 2: Stylometrics  --> stylometric_score (0-1)
  |
  v
Confidence Scorer  (0.6 x llm + 0.4 x stylo) --> attribution + confidence
  |
  v
Label Generator --> label text
  |
  |-- Storage (submissions.json)
  |-- Audit Log (audit_log.json)
  +-- API Response {content_id, attribution, confidence, label, signals}

APPEAL FLOW
-----------
POST /appeal {content_id, creator_reasoning}
  --> Lookup submission --404--> Client
  --> Update status to "under_review"
  --> Audit Log (event_type: "appeal")
  --> API Response {appeal_id, status, message}
```

## Detection Signals

### Signal 1: LLM-Based Classification (Groq)

**What it measures:** Holistic semantic and stylistic coherence. The model (`llama-3.3-70b-versatile`) evaluates tone, sentence flow, repetition, register consistency, and the presence of AI-typical filler phrases ("it is important to note," "furthermore," "in conclusion"). It returns a structured `ai_probability` score.

**Why it captures AI writing:** AI-generated text is grammatically perfect and tonally consistent. It avoids slang, typos, and abrupt register shifts. The LLM can detect these holistic patterns as a trained observer.

**What it misses:** Polished formal human writing can look AI-like. AI text prompted to sound casual can look human. Very short texts (under 30 words) give insufficient signal.

---

### Signal 2: Stylometric Heuristics (Pure Python)

**What it measures:** Three structural metrics combined into a single score:

| Metric                                           | What high values mean                      |
| ------------------------------------------------ | ------------------------------------------ |
| Sentence-length coefficient of variation (SL-CV) | Low variance (uniform sentences) = AI-like |
| Type-token ratio (TTR)                           | Low vocabulary diversity = AI-like         |
| Average word length                              | Longer formal words = AI-like              |

**Why it captures AI writing:** AI text follows more uniform statistical patterns at the structural level. Human writing varies more in rhythm, word choice, and sentence construction because it reflects individual thought rather than statistical prediction.

**What it misses:** Formal academic writing by non-native English speakers scores AI-like on all three metrics. Poetry with intentional repetition scores AI-like on TTR. Lightly edited AI text can retain structural uniformity even when semantics change.

---

### Why Both Signals Together

The signals are independent - one evaluates meaning, the other ignores meaning entirely. When both agree, confidence is high. When they disagree, the system reports uncertainty rather than forcing a verdict. The LLM signal has 60% weight because it captures semantic patterns the structural metrics cannot.

## Confidence Scoring

### Formula

```
combined_score = (0.6 x llm_score) + (0.4 x stylometric_score)
```

If the Groq API is unavailable, the system falls back to stylometrics only (`combined_score = stylometric_score`) and sets `"llm_available": false` in the response.

### Thresholds

| Score range  | Attribution    | Reasoning                                      |
| ------------ | -------------- | ---------------------------------------------- |
| > 0.70       | `likely_ai`    | Both signals lean AI with sufficient margin    |
| < 0.35       | `likely_human` | Both signals lean human with sufficient margin |
| 0.35 to 0.70 | `uncertain`    | Signals diverge or margin is too small         |

The thresholds are **asymmetric** (0.35 vs 0.70). A false positive - labeling a human creator's work as AI - is more harmful than a false negative on a creative platform. The wider uncertain zone gives human creators benefit of the doubt at the margins.

### How Confidence Is Reported

- `likely_ai`: `confidence = combined_score`
- `likely_human`: `confidence = 1.0 - combined_score`
- `uncertain`: `confidence = 0.5 + abs(combined_score - 0.525)` - reflects how close to the edge of the uncertain zone the result sits

A score of 0.51 produces ~51% confidence; a score of 0.68 produces ~65% confidence. A non-technical user reading the label sees a meaningful difference between these.

### Example Submissions with Different Confidence Scores

**High-confidence example (AI-generated text):**

```
Input: "Artificial intelligence represents a transformative paradigm shift in
modern society. It is important to note that while the benefits of AI are
numerous, it is equally essential to consider the ethical implications..."

llm_score:          0.90
stylometric_score:  0.519
combined_score:     0.748
attribution:        likely_ai
confidence:         0.748  (75%)
```

**Lower-confidence example (formal human writing - uncertain zone):**

```
Input: "The relationship between monetary policy and asset price inflation has
been extensively studied in the literature. Central banks face a fundamental
tension between their mandate for price stability..."

llm_score:          0.80
stylometric_score:  0.528
combined_score:     0.691
attribution:        uncertain
confidence:         0.666  (67%)
```

The high-confidence case scores 75% and receives a `likely_ai` label with an appeal pathway. The uncertain case scores 67% but falls below the 0.70 threshold, receiving the `uncertain` label - the content is not flagged and the creator is told no action has been taken.

## Transparency Label

Three verbatim variants. `{pct}` is replaced with `round(confidence * 100)`.

### Variant 1: High-Confidence AI (`likely_ai`, score > 0.70)

> "This content was likely generated with AI assistance. Our analysis detected patterns consistent with AI-generated writing with {pct}% confidence. If you are the creator and believe this is incorrect, you can submit an appeal for human review."

### Variant 2: High-Confidence Human (`likely_human`, score < 0.35)

> "This content appears to be human-written. Our analysis found natural variation in writing style consistent with human authorship, with {pct}% confidence. No attribution concern has been flagged."

### Variant 3: Uncertain (0.35 to 0.70)

> "Our system could not confidently determine whether this content is human-written or AI-generated ({pct}% confidence). The content has not been flagged. If you are the creator and have concerns, you may submit an appeal for human review."

**Design notes:**

- Variants 1 and 3 mention the appeal pathway because those are the cases where a human creator might be harmed.
- Variant 2 omits it - a human verdict requires no appeal.
- "The content has not been flagged" in Variant 3 reassures creators in the uncertain zone that no punitive action has been taken against their work.

## Rate Limiting

**Limits applied to `POST /submit`:** `10 per minute; 100 per day`

**Reasoning:**

- A legitimate creator submitting their own work does at most 2-3 submissions in a sitting (draft, revision, final). 10 per minute is generous for legitimate use while blocking scripted floods.
- 100 per day comfortably covers a prolific creator submitting 10-20 pieces per day with retries.
- An adversary flooding the system would be blocked after 10 rapid requests. Since the 429 response does not reveal scoring details, probing the classification boundary through flooding provides no useful information.
- The Groq free tier has its own quota; the rate limiter also protects against accidentally burning through it via rapid automated requests.

## Known Limitations

### 1. Non-Native and Formal Human Writing Triggers False Positives

A non-native English speaker who writes in careful, structured prose will score high on all three stylometric metrics (low sentence-length variance, low TTR from precise word choices, long formal words) and the LLM may also flag the formal register as AI-like. This is the highest-risk false-positive scenario and is explicitly handled by the appeal pathway. The asymmetric threshold (flagging only above 0.70, not above 0.50) reduces but does not eliminate this risk.

### 2. Short Text Reduces Stylometric Reliability

Texts under ~50 words give the stylometric signal insufficient data. The sentence-length CV is undefined for a single sentence (returns 0.5), and TTR is inflated for short texts because few words repeat at that length. For submissions under 30 words, the system effectively relies on the LLM signal only.

## Spec Reflection

**One way the spec helped:** Writing the three label variants in `planning.md` before any code forced a decision about the asymmetric uncertainty zone. The phrase "The content has not been flagged" in Variant 3 came directly from thinking through the appeal use case: a creator in the uncertain zone should not be anxious, because no action has been taken. That design decision would have been easy to overlook without pre-writing the label text.

**One way implementation diverged from the spec:** The planning document listed "punctuation density" as the third stylometric metric. During calibration, all four test inputs scored identically on punctuation density (all fell below the 4% threshold for short texts, returning 0.7 uniformly). The metric produced no signal. It was replaced with average word length, which correctly separates casual human vocabulary from AI formal vocabulary and produced meaningful spread across the test battery.

## AI Usage

### Instance 1: Detection Pipeline and Groq Client Scaffolding

**Directed:** Provided the detection signals section from `planning.md` and the architecture diagram, then asked for the Flask app skeleton with a `POST /submit` route stub and a `classify_text()` function that returns only a structured JSON response from the Groq API.

**What was generated:** A working `groq_client.py` that called the API and parsed the response.

**What was revised:** The generated function did not handle the case where the model wraps its JSON response in markdown code fences. Added a stripping block after observing this behavior in manual testing. Also added the `llm_available` flag to `DetectionResult` and corrected the weighting from 50/50 to 60/40 as specified in planning.md.

### Instance 2: Confidence Scoring and Label Generation

**Directed:** Provided the uncertainty representation section and the three verbatim label variants from `planning.md`, then asked for the confidence scoring function and the label generator.

**What was generated:** A confidence scorer that mapped combined scores to attributions and a label function that returned label text.

**What was revised:** The generated confidence value for `uncertain` results was the raw `combined_score`, which would show "63% confidence" for a score of 0.63 - misleading because that score is well within the uncertain zone, not particularly confident. Replaced with `0.5 + abs(combined_score - 0.525)` to reflect actual distance from the zone midpoint. Also added the phrase "The content has not been flagged" to Variant 3, which was absent in the generated version.

## API Reference

### `POST /submit`

Classify a piece of text for attribution.

**Request:**

```json
{ "text": "string (required)", "creator_id": "string (required)" }
```

**Response 200:**

```json
{
  "content_id": "uuid",
  "attribution": "likely_ai | likely_human | uncertain",
  "confidence": 0.748,
  "label": "verbatim label text",
  "signals": {
    "llm_score": 0.9,
    "stylometric_score": 0.519,
    "llm_available": true
  },
  "status": "classified",
  "timestamp": "ISO 8601"
}
```

**Errors:** `400` missing field - `429` rate limit - `500` classification failure

---

### `POST /appeal`

Contest a classification decision.

**Request:**

```json
{ "content_id": "uuid (required)", "creator_reasoning": "string (required)" }
```

**Response 200:**

```json
{
  "appeal_id": "uuid",
  "content_id": "uuid",
  "status": "under_review",
  "original_attribution": "likely_ai",
  "message": "Your appeal has been received and the content is now under review."
}
```

**Errors:** `400` missing field - `404` content_id not found

---

### `GET /log`

Return recent audit log entries.

**Query params:** `limit` (int, default 50)

**Response 200:**

```json
{ "entries": [...], "count": 7 }
```
